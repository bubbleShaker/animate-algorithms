# 動画の目視レビュー手法（フレーム抽出 → 確認 → 自律修正）

> Manim 等で生成した mp4 の「見た目の違和感」を、Claude が**動画を直接見て**特定し、
> ソースを直して再レンダリングし、**直ったかどうかまで自分で確認する**ための手順。
> 2026-06-24 に bubble_sort のバー底辺ズレ不具合を直した実例から一般化したもの。

## いつ使うか

- 「動画に違和感がある」「なんか変」など、**ログやテストには出ない見た目の不具合**を直したいとき。
- アニメーションの整列・配置・色・タイミングなど、**最終フレームや途中フレームを見ないと判断できない**問題。

## なぜこの手法か（代替案との比較）

- Claude は mp4 を再生できないが、**静止画 (PNG) なら直接「見る」ことができる**。
  → 動画を ffmpeg でフレーム画像に分解すれば、目視レビューが可能になる。
- 代替案「ユーザーに口頭で症状を説明してもらう」より、**Claude が一次情報（実画像）に当たれる**ぶん誤診が減る。
- 代替案「ソースだけ読んで推測」より、**実際の描画結果と突き合わせられる**ので原因特定が速い。

## 環境前提（このリポジトリの場合）

- **Windows には ffmpeg も Manim も無い**。どちらも **WSL 側**にある。
  - ffmpeg: `/usr/bin/ffmpeg`
  - Manim: pipx 管理 → `~/.local/share/pipx/venvs/manim/bin/manim`（PATH に無いのでフルパス推奨）
- Windows パスは WSL から `/mnt/c/...` で参照する。
- フレーム画像は **スクラッチパッド（Windows 側の一時領域）** に出すと、Claude の Read ツールでそのまま読める。
  - 関連: [knowledge/manim-setup-windows-wsl.md](../knowledge/manim-setup-windows-wsl.md)

## 手順

### 1. 動画の長さを確認し、フレームを抽出する

```bash
VID="/mnt/c/Users/moai0/git/animate-algorithms/media/videos/<scene>/480p15/<Scene>.mp4"
OUT="<スクラッチパッドの WSL パス>/frames"

wsl.exe -e bash -lc "
  ffprobe -v error -show_entries format=duration -of default=nw=1:nk=1 '$VID'  # 秒数
  mkdir -p '$OUT'; rm -f '$OUT'/*.png
  ffmpeg -hide_banner -loglevel error -i '$VID' -vf fps=2 '$OUT/f_%03d.png'    # 2fps で抽出
  ls '$OUT' | wc -l
"
```

- `fps=2` は「1秒あたり2枚」。短い動画なら 2〜4、長ければ 1 など、**フレーム数が 20〜40 枚程度**に収まる値にする。
- `ffprobe` … 動画のメタ情報（長さ等）を取る ffmpeg 付属ツール。抽出枚数の見積もりに使う。

### 2. フレームを Read で「見て」違和感を特定する

- まず **間引いて**（例: 1, 4, 7, 10, … 番目）全体像を掴む → 怪しい区間を**ピンポイントで**追加で読む。
- 特に **最終フレーム**は「完成状態が正しいか」の判定に効く（整列・配置の最終結果が一目で分かる）。
- 全フレームを一度に読むとコストが高いので、**当たりを付けてから絞る**。

### 3. 原因をソースで特定して直す

- フレームで見えた症状（例:「底辺がバラつく」）から、対応するコード（例: 位置移動ロジック）を当てる。
- 修正は最小限に。**なぜ直るのか**をコメントで残す。

### 4. 再レンダリングする

```bash
wsl.exe -e bash -lc '
  cd /mnt/c/Users/moai0/git/animate-algorithms
  MANIM=~/.local/share/pipx/venvs/manim/bin/manim
  "$MANIM" -ql scenes/<scene>.py <Scene> 2>&1 | tail -15
'
```

- `-ql` = 低画質プレビュー（480p / 15fps）。確認用には十分速い。本番は `-qh` 等に上げる。
- `-p`（自動再生）は付けない。CLI 環境で開けないため。

### 5. 直ったか再確認する（重要）

- **手順1〜2をもう一度**、別フォルダ（例 `frames2/`）に抽出して見る。
- 修正前のフレームと**同じ時刻・同じ構図**を見比べると、直ったことを客観的に確認できる。
- ここまでやって初めて「自律的に直した」と言える。直ったフレームを根拠として提示する。

## 実例（2026-06-24 bubble_sort）

- **症状**: スワップのたびにバーの底辺がバラバラに浮き、整列完了時も高さ順に見えない。
- **原因**: `_swap_bars` が `get_center()` ↔ `move_to(center)` で**中心ごと**交換 → 高さの違うバーは中心の高さが異なるため底辺がズレた。
- **修正**: `get_x()` / `set_x()` で**横位置 (x) だけ**を交換し、底辺 (y) は固定。
- **確認**: 再抽出フレームで底辺が揃い、最終フレームが `1,2,3,5,6,9` のきれいな階段状になった。

## 落とし穴メモ

- `fps` を上げすぎると枚数が爆発し、Read コストが膨らむ。先に `ffprobe` で長さを見る。
- Windows の PATH に ffmpeg/manim は無い。**必ず WSL 側**で実行する。
- フレームは**スクラッチパッド（Windows 側一時領域）**に出す。WSL のホーム配下に出すと Read ツールから読みにくい。
- manim は pipx 管理で PATH に出ていないことがある。**フルパス**で叩くのが確実。
